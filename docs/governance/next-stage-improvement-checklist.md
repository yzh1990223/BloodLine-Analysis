# 下一阶段改进清单

这份清单用于说明 BloodLine Analysis 的 Vibe Coding 体系下一步最值得补什么。

目标不是一口气变成重型治理平台，而是在当前基础上，从“成熟单仓治理”继续提升到“更稳定、更易复制”的下一阶段。

## 当前成熟度判断

当前体系已经具备：

- 仓库事实层
- 本地门禁层
- 远端兜底层
- 方法层与治理层的清晰边界

整体已经明显高于普通单仓项目，但还没有到团队级重治理的程度。

## 优先级 P1：现在最值得做

### 1. 增加轻量提交辅助入口

当前仓库已经有：

- `docs/governance/commit-workflow.md`

但还没有一个更直接的“提交辅助入口”，例如：

- `scripts/hooks/commit-helper.sh`
- 或专门的提交操作说明文档入口

目标：

- 更容易判断什么时候应拆分提交
- 更容易区分代码批次、文档批次、治理批次

### 2. 继续增强文档映射规则

当前 `doc-sync-check.sh` 已经有路径映射意识，但仍以提醒为主。

下一步可继续细化：

- 改 API → 提醒 README + UAT
- 改图模型 / 核心服务 → 提醒 spec + UAT
- 改治理脚本 / CI → 提醒 governance docs

目标：

- 让文档联动更具体、更可执行

### 3. 正式固定 AI hooks 的定位

当前已经有：

- `docs/governance/ai-hook-integration.md`

下一步应继续坚持这条边界：

- AI hooks 是事件辅助层
- 不是第二套 workflow

目标：

- 避免未来治理文档和 `superpowers` 方法层打架

## 优先级 P2：适合下一阶段继续补

### 4. 给 governance smoke 增加行为级验证

当前 smoke 主要检查：

- 文件存在
- 脚本语法
- 关键专项检查可运行

下一步可以继续补：

- hook 安装是否成功
- 某些脚本失败分支是否符合预期

### 5. 让 CI 与本地门禁继续收敛

当前已经比较接近，但仍可以继续收敛到：

- 本地门禁跑什么
- CI 就尽量跑什么

目标：

- 避免本地和远端两套标准漂移

### 6. 给治理文档增加角色化入口

例如把治理文档分成：

- 新开发者先读什么
- AI 先读什么
- 新项目复制模板时先读什么

目标：

- 降低上手成本

## 优先级 P3：多人协作时再考虑

### 7. 审查状态机制

例如：

- review fingerprint
- `.review-passed`
- review 失效检测

这类能力更适合多人协作或高频提交流程。

### 8. 任务 / 路线图联动

例如：

- 活跃任务模板
- issue / roadmap / 文档联动
- 阶段状态 SSOT

### 9. 经验沉淀闭环

目标是形成：

- bug
- 复盘
- 规则
- hook / 文档 / smoke

之间的自动回灌链路。

## 不建议现在做的事

当前阶段不建议直接上：

- 复杂 review 凭证体系
- 外部通知系统
- 多层任务状态机
- 大而全的自动化脚本平台

原因很简单：会明显增加治理成本，且对当前仓库阶段收益不够高。

## 总结

当前最推荐的方向不是“继续堆更多规则”，而是继续强化这三件事：

1. 提交辅助
2. 文档映射
3. AI hooks 边界稳定化

如果把这三件事继续做好，当前体系就能从“成熟单仓治理”继续往“高可复制、高可维护治理”迈一步。
