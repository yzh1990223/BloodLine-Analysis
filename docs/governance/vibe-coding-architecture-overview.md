# Vibe Coding 架构总览

这份文档描述 BloodLine Analysis 当前的整体 Vibe Coding 体系。

目标不是再发明一套抽象方法论，而是把这个仓库里已经落地的协作结构说清楚：

- 规则放在哪里
- 方法放在哪里
- 过程提醒放在哪里
- 入库门禁放在哪里
- 远端兜底放在哪里

## 总体分层

当前体系可以理解为 5 段闭环：

1. 项目事实层
2. 方法层
3. 过程辅助层
4. 本地门禁层
5. 远端兜底层

## 1. 项目事实层

职责：

- 定义这个仓库是什么
- 定义当前已支持与未支持边界
- 定义必须执行的验证命令
- 定义文档联动规则与禁止事项

核心载体：

- `AGENTS.md`
- `backend/AGENTS.md`
- `frontend/AGENTS.md`
- `README.md`
- `docs/governance/`

这层的价值在于让 AI 和人都先拿到同一份高优先级上下文，降低“按经验脑补仓库事实”的风险。

## 2. 方法层

职责：

- 决定任务应该怎么推进
- 决定什么时候先设计、先调试、先验证
- 决定如何拆解问题与收口结果

核心载体：

- `superpowers`

这层不是仓库文件本身，而是 AI 工作方法。它负责“怎么做事”，而不是“仓库事实是什么”。

## 3. 过程辅助层

职责：

- 在开发过程的特定事件点提醒和守卫
- 尽早提示风险，而不是等到 commit / push 再发现

核心载体：

- `scripts/hooks/ai-session/`
- `docs/governance/ai-hook-integration.md`

当前状态：

- 已有脚本骨架
- 已有接入说明
- 尚未和固定客户端强绑定

这层应被视为“事件辅助层”，不是第二套完整工作流系统。

## 4. 本地门禁层

职责：

- 在代码进入本地提交历史之前做客观检查
- 把关键规则从“建议”变成“门禁”

核心载体：

- `lefthook.yml`
- `scripts/hooks/`

当前已覆盖：

- `pre-commit`
- `commit-msg`
- `post-commit`
- `pre-push`

当前关键能力：

- 受保护文件检查
- 提交粒度检查
- 文档同步提醒
- 模型 / 迁移联动提醒
- API / 前后端同步检查
- push 前测试、构建和治理 smoke

## 5. 远端兜底层

职责：

- 本地漏掉时，在远端 CI 再做一次兜底

核心载体：

- `.github/workflows/ci.yml`

当前已跑：

- governance smoke
- backend tests
- frontend tests
- frontend build

## 当前最重要的边界

为了避免重复和冲突，当前体系遵守这条边界：

- `AGENTS.md`
  - 定义仓库事实与仓库规则
- `superpowers`
  - 定义工作方法
- AI hooks
  - 负责事件提醒和轻守卫
- Git hooks / CI
  - 负责客观门禁

这条边界是当前体系能够持续演进而不失控的关键。

## 当前优势

这套体系当前最强的地方不在某一个 hook，而在于层和层之间已经形成了闭环：

- 规则不是只有文档
- 门禁不是只有脚本
- CI 不是另一套平行标准
- AI 方法论也没有和仓库规则混写在一个文件里

这意味着它已经明显高于“只有 README 和测试命令”的普通项目治理水平。

## 当前限制

当前体系仍然不是重型团队治理平台，主要限制包括：

- 没有审查状态凭证体系
- 没有任务路线图联动系统
- AI hooks 还未真正挂入固定客户端
- 文档同步仍以提醒型规则为主

这些限制并不表示体系无效，只表示它当前更适合单仓、轻团队、快速演进阶段。
