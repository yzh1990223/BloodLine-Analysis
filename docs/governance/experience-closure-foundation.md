# 经验沉淀闭环基线

这份文档说明经验沉淀闭环如何接入 BloodLine Analysis 当前治理体系。

## 目标

经验体系的目标不是“多一个 docs 目录”，而是让项目中的经验可以稳定完成：

`记录 -> 提炼 -> 回灌`

## 当前范围

第一版覆盖：

- bug / 故障 / 误判经验
- 实现与设计经验
- UAT / 部署 / 运行经验
- 治理与 AI 协作经验

## 当前结构

经验内容统一放在：

- `docs/experience/`

核心入口包括：

- `docs/experience/README.md`
- `docs/experience/indexes/experience-index.md`
- `docs/experience/digests/`

## 和现有治理的关系

经验闭环并不是平行于治理体系之外的新层，而是当前治理体系的一部分：

- `AGENTS.md`
  - 提供项目级入口规则
- `README.md`
  - 提供使用入口
- `tests/governance_smoke.sh`
  - 检查经验核心文档存在
- 阶段总结与下一阶段清单
  - 承接经验回灌后的演进方向

## 什么经验值得沉淀

推荐沉淀这些经验：

- 修掉一个真实 bug 后
- 发现一个反复出现的误判模式后
- 做出重要设计取舍后
- UAT / 部署中出现可复用经验后
- 治理体系本身暴露缺口后

## 回灌原则

一条经验只有在明确了回灌动作之后，才算真正有价值。

回灌目标可以是：

- 文档
- 测试
- hooks / CI
- `AGENTS.md`
- backlog / 下一阶段需求

## 第一版策略

第一版不做：

- 复杂评分
- 自动状态机
- 数据库存储
- 自动化经验平台

第一版先确保：

- 有统一目录
- 有统一模板
- 有索引
- 有真实样例
- 有治理入口

## 第二版补充

第二版在 `post-commit` 增加了轻量提醒器。

它的定位是：

- 在提交成功后提示“这次是否值得沉淀经验”
- 给出建议经验类型
- 指向经验入口文档

第二版仍然坚持：

- 只提醒
- 不阻断
- 不自动创建经验草稿
