# MySQL 元数据接入设计

## 1. 背景

当前 `BloodLine Analysis` 已经具备：

- `.repo` 解析
- Java 源码静态 SQL 解析
- 对象级 / 表级血缘
- 影响分析
- 闭环分析

但当前扫描流程中的 `mysql_dsn` 仍然只是预留字段，尚未真正参与任何元数据读取或对象归一化。

这带来几个现实问题：

- 同名对象的归一化仍然偏弱
- 无法稳定区分真实表与视图
- 前端对象详情缺少来自数据库的补充信息
- 后续字段级血缘缺少元数据基础

因此，`#3 epic: MVP2 - MySQL 元数据接入` 的第一阶段目标，是让 MySQL 元数据真正进入当前扫描链路，并提升对象归一化与展示质量。

## 2. 目标

本轮设计目标是：

1. 在扫描时真实消费 `mysql_dsn`
2. 读取指定库范围内的 MySQL 元数据
3. 将元数据用于对象归一化与合并
4. 区分 `data_table` 与 `table_view`
5. 为对象节点补充字段元信息摘要
6. 为后续字段级血缘保留数据基础

## 3. 非目标

本轮明确不做：

- 字段级血缘推导
- 存储过程解析
- 外键关系建图
- 元数据缓存子系统
- 异步元数据同步任务
- 全实例无约束扫描

## 4. 核心边界

### 4.1 元数据加载方式

第一版采用：

- 扫描内同步加载元数据

也就是在调用扫描接口时，如果提供 `mysql_dsn`，扫描流程会直接连接 MySQL 读取元数据。

设计上保留边界：

- 元数据读取逻辑独立成单独模块
- 后续可演进为“先同步元数据、再扫描读取缓存”的模式

### 4.2 元数据的作用范围

本轮元数据不仅用于展示，还必须参与：

- 对象归一化
- 表名补全
- 对象合并

### 4.3 合并粒度

本轮先做到：

- 表级合并
- 视图区分

即：

- 同一实例、同一业务库、同一对象名可统一成同一对象节点
- 视图对象单独标记为 `table_view`

### 4.4 读取范围

本轮采用库白名单策略：

- 新增可选输入 `metadata_databases`
- 若传入，则只读取这些库
- 若未传入，则默认回退到 DSN 中的默认库

不做：

- 默认读取整个实例的全部可见库

## 5. 输入设计

扫描请求体建议扩展为：

- `repo_path`
- `java_source_root`
- `mysql_dsn`
- `metadata_databases`

其中：

- `mysql_dsn`
  - MySQL 连接串
- `metadata_databases`
  - 可选字符串数组
  - 控制要读取的库白名单

### 5.1 输入规则

规则如下：

1. 若没有 `mysql_dsn`
   - 不执行元数据读取
   - 扫描行为保持与当前版本一致
2. 若有 `mysql_dsn` 且有 `metadata_databases`
   - 只读取指定库
3. 若有 `mysql_dsn` 但无 `metadata_databases`
   - 只读取 DSN 默认库

## 6. 模块设计

新增独立模块：

- `backend/src/bloodline_api/connectors/mysql_metadata.py`

模块职责：

1. 建立 MySQL 连接
2. 按白名单读取 `information_schema`
3. 返回统一的元数据中间结构

建议返回结构包含：

- 数据库名
- 对象名
- 是否为视图
- 字段列表
- 表注释
- 字段注释

### 6.1 为什么必须独立模块化

这样做的价值是：

- 避免把连接与查询逻辑混进 `lineage_query.py`
- 方便后续演进为缓存式加载
- 方便单元测试
- 方便后续扩展到更多 metadata 来源

## 7. 扫描流程接入

当前 `scan_from_inputs()` 的逻辑需要演进为：

1. 解析 `.repo`
2. 解析 Java 源码
3. 根据 `mysql_dsn` 与 `metadata_databases` 读取 MySQL 元数据
4. 对 repo / Java 解析出的对象进行归一化
5. 写入 `nodes / edges`
6. 再派生 `FLOWS_TO`

### 7.1 关键原则

- 元数据读取失败不应默默吞掉
- 但失败信息必须标准化返回
- 不能因为元数据模块接入而破坏当前无 `mysql_dsn` 的扫描路径

## 8. 归一化与对象合并规则

### 8.1 目标

本轮目标不是“最大化合并”，而是“优先避免错并”。

### 8.2 规则

1. 如果 repo / Java 解析出的对象能明确命中：
   - `database.table`
   - 则合并到对应 metadata 对象
2. 如果对象只有裸表名
   - 先尝试在白名单库中唯一匹配
   - 若唯一，则合并
3. 如果裸表名在多个库中同时存在
   - 第一版不强行合并
   - 保守保留原对象，避免错并
4. 如果 metadata 显示对象为 view
   - 将对象类型标记为 `table_view`
5. 其余稳定命中的数据库对象
   - 标记为 `data_table`

### 8.3 不做的规则

本轮不做：

- 基于字段相似度猜测表归属
- 基于注释或模糊命名做智能合并
- 跨环境对象自动映射

## 9. 数据模型设计

第一版不强制新增独立 metadata 表，而是复用当前 `nodes.payload` 承载补充信息。

建议在对象节点 payload 中补充：

- `database`
- `table_name`
- `is_view`
- `columns`
- `comment`
- `metadata_source`

其中：

- `metadata_source`
  - 当前可写为 `mysql_information_schema`

### 9.1 为什么先不拆表

当前更重要的是先跑通能力闭环：

- 读取
- 归一化
- 展示

如果第一版就引入独立 metadata 表，会显著增加迁移、查询和同步复杂度。

## 10. 前端影响

本轮前端只做轻增强，不新增复杂页面。

建议增强点：

1. 对象详情页
   - 展示库名
   - 展示对象类型是否为视图
   - 展示字段数量或字段摘要
2. 搜索结果页
   - 提高对象类型准确性
3. 现有图页面
   - 不新增字段级图
   - 不改变主交互逻辑

## 11. API 影响

### 11.1 扫描接口

`POST /api/scan` 需要真正消费：

- `mysql_dsn`
- `metadata_databases`

### 11.2 查询接口

现有对象查询接口无需大改，但返回 payload 中会包含更多 metadata 字段。

## 12. 风险与应对

### 12.1 误合并风险

风险：

- 跨库同名对象被误并

应对：

- 裸表名遇到多库命中时保守不合并

### 12.2 性能风险

风险：

- 元数据读取会拖慢扫描

应对：

- 强制使用库白名单或默认库回退
- 不读取无关系统库

### 12.3 可用性风险

风险：

- 数据库权限或连通性问题导致扫描失败

应对：

- 错误信息标准化
- 在扫描结果中保留明确失败原因

## 13. 测试策略

建议最少覆盖：

1. `mysql_dsn` 缺失时扫描保持旧行为
2. 白名单库过滤正确
3. 表 / 视图区分正确
4. 同库同名对象可合并
5. 多库同名对象不误并
6. 查询结果中能看到 metadata 补充信息

## 14. 子任务建议

建议将 `#3` 拆成以下第一批 Task：

1. `task: 设计 metadata connector 与连接边界`
2. `task: 读取 information_schema 库表字段元数据`
3. `task: 扩展后端 schema / persistence 支持 metadata payload`
4. `task: 扫描流程接入 mysql metadata`
5. `task: 前端展示元数据补充信息`
6. `task: 更新 README / 部署文档 / UAT 手册`
7. `task: 增加 metadata 回归测试`

## 15. 结论

`#3` 第一阶段最合理的落地方式是：

- 扫描内同步读取 MySQL 元数据
- 使用独立 metadata loader 模块承载连接与读取逻辑
- 先做到表级对象合并与视图区分
- 用库白名单控制读取范围
- 不把第一版做成重型元数据平台

这样既能快速把 `mysql_dsn` 变成真实能力，又能给后续字段级血缘和元数据缓存式演进留下清晰边界。
