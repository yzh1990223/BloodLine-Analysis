# 最近一次扫描失败汇总页设计

日期：2026-04-01  
关联 Issue：`#42`

## 1. 背景

当前 BloodLine Analysis 的扫描链路已经覆盖：

- Kettle `.repo` / repository XML 解析
- Java 源码解析
- MySQL metadata 加载与视图定义解析

但这些环节里的失败信息仍然是分散的：

- 有些失败只出现在运行日志
- 有些失败只保存在对象详情页，例如视图解析失败
- 有些失败会直接以接口报错暴露，但不会进入统一的排查视图

这导致用户在扫描后排查问题时，必须在日志、对象详情和页面报错之间来回切换，缺少一个统一入口。

## 2. 目标

本轮目标是新增一个单独的“最近一次扫描失败汇总页”，满足以下要求：

1. 默认只展示最近一次扫描的失败项
2. 以来源类型分组展示：
   - `Kettle`
   - `Java`
   - `MySQL Metadata`
3. 以文件为聚合单位展示失败
4. 每条失败至少展示：
   - 文件路径
   - 失败类型
   - 失败原因
   - 相关对象或 SQL 片段摘要

## 3. 非目标

本轮不做：

- 历史扫描切换与对比
- 全量诊断中心
- 普通 warning 汇总
- 自动修复建议
- 失败项搜索、过滤、导出

## 4. 设计原则

### 4.1 失败信息必须在扫描时持久化

失败汇总页不能依赖临时日志，因为：

- 日志不可结构化消费
- 页面刷新后日志不可追溯
- 不利于后续扩展成历史扫描诊断

因此，本轮以“扫描时落库”为基础。

### 4.2 页面只看最近一次扫描，但数据结构为后续历史扩展留接口

用户当前只要求“最近一次扫描失败汇总”，所以页面和接口默认都围绕 latest scan 展开。  
但存储层按 `scan_run_id` 建模，方便后续扩展到历史扫描。

### 4.3 文件级聚合优先于对象级堆叠

用户希望默认按文件看问题，而不是一上来被大量对象级失败淹没。  
因此返回结构会先按来源分组，再按文件聚合，文件下再挂失败明细。

## 5. 失败来源定义

### 5.1 Kettle

收录范围：

- `.repo` / repository XML 中单条 SQL 解析失败
- Kettle SQL 片段抽取失败
- 其他不应中断整次扫描、但会导致对象血缘缺失的解析失败

说明：

- 不把“正常无法识别的空配置”当成失败
- 只记录明确影响结果质量的解析失败

### 5.2 Java

收录范围：

- Java 字符串 SQL 解析失败
- 注解 SQL 解析失败
- XML Mapper SQL 解析失败
- Java API / 方法级事实归并中被明确标记为失败的项

说明：

- 本轮仍不收录普通 unresolved 调用关系，除非它们被升级成结构化失败项

### 5.3 MySQL Metadata

收录范围：

- 视图 `VIEW_DEFINITION` 解析失败
- 单个 metadata 对象级解析失败但扫描整体成功的情况

说明：

- 连接级失败仍按现有接口错误直返，不进入“最近一次扫描失败汇总”
- 汇总页聚焦“扫描已完成，但存在局部失败”的情况

## 6. 数据模型设计

新增一张失败记录表：

- `scan_failures`

字段建议：

- `id`
- `scan_run_id`
- `source_type`
  - `kettle`
  - `java`
  - `metadata`
- `file_path`
- `failure_type`
- `message`
- `object_key`
- `sql_snippet`
- `created_at`

字段说明：

- `file_path`
  - 作为文件级聚合主键之一
- `failure_type`
  - 例如：
    - `sql_parse_failed`
    - `view_definition_parse_failed`
    - `java_annotation_sql_parse_failed`
- `message`
  - 面向用户的失败原因
- `object_key`
  - 可为空，用于关联相关对象
- `sql_snippet`
  - 可为空，保留失败时的 SQL/定义摘要

## 7. 后端流程设计

### 7.1 扫描开始前

每次新扫描开始时：

- 创建 `scan_run`
- 后续所有失败记录都挂到该 `scan_run_id`

### 7.2 扫描过程中

在各解析阶段，把当前已经出现的“可结构化失败”写入 `scan_failures`。

本轮建议最先接入的写入点：

1. SQL 抽取器返回错误时
2. Java/MyBatis SQL 抽取失败时
3. 视图定义解析失败时

### 7.3 扫描完成后

扫描仍可保持 `completed`，即使存在 `scan_failures`。  
也就是说，失败汇总页表达的是“本轮扫描已完成，但有局部失败项”。

## 8. API 设计

新增接口：

- `GET /api/scan-runs/latest/failures`

返回结构建议：

- `scan_run`
  - `id`
  - `status`
  - `started_at`
  - `finished_at`
- `summary`
  - `group_count`
  - `file_count`
  - `failure_count`
  - `kettle_file_count`
  - `java_file_count`
  - `metadata_file_count`
- `groups`
  - `source_type`
  - `label`
  - `file_count`
  - `failure_count`
  - `files`
    - `file_path`
    - `failure_count`
    - `items`
      - `failure_type`
      - `message`
      - `object_key`
      - `sql_snippet`

说明：

- 接口直接返回前端可消费的聚合结构
- 不要求前端自己做复杂 group-by

## 9. 前端页面设计

新增页面：

- `/scan-failures`

入口建议：

- 首页扫描面板附近增加入口：
  - `查看最近扫描失败汇总`

页面结构：

1. 顶部摘要区
   - 最近扫描时间
   - 总失败文件数
   - 总失败项数
   - Kettle / Java / Metadata 文件数

2. 分组区
   - `Kettle`
   - `Java`
   - `MySQL Metadata`

3. 文件折叠区
   - 默认按文件收拢展示
   - 每个文件显示：
     - 文件路径
     - 失败条数

4. 失败详情区
   - 展开文件后显示具体失败项
   - 每条显示：
     - 失败类型
     - 失败原因
     - 相关对象
     - SQL/定义摘要

## 10. 与现有页面的关系

### 10.1 首页

首页只提供入口和最近扫描摘要，不承载完整失败列表。

### 10.2 对象详情页

像视图详情页里现有的“视图解析失败信息”仍然保留。  
失败汇总页不是替代它，而是提供一个全局入口。

## 11. 错误处理与空态

### 11.1 没有扫描记录

页面显示：

- `暂无扫描记录`

### 11.2 最近一次扫描无失败项

页面显示：

- `最近一次扫描未记录到失败项`

### 11.3 接口请求失败

页面显示统一错误文案，并保留返回总览入口。

## 12. 测试策略

后端需要覆盖：

1. 最近一次扫描失败汇总接口返回空态
2. 多来源失败项按文件聚合正确
3. 只返回最近一次扫描的失败项
4. 视图失败项能正确出现在 `metadata` 分组

前端需要覆盖：

1. 页面空态
2. 分组渲染
3. 文件级折叠展示
4. 失败详情展开显示

## 13. 风险与权衡

### 风险 1：当前部分失败点尚未结构化

不是所有失败现在都已经能以结构化形式拿到。  
因此第一版汇总页可能先覆盖：

- metadata 视图解析失败
- SQL 抽取失败

再逐步扩展到更多 Java / Kettle 场景。

### 风险 2：失败记录过多导致页面过载

文件级聚合能缓解这一点。  
本轮先不做搜索过滤，后续如果失败量继续扩大，再增加筛选能力。

## 14. 推荐实施顺序

1. 新增 `scan_failures` 表与模型
2. 在 metadata 视图解析失败处先接入失败记录
3. 在 SQL 抽取失败路径接入 Kettle / Java 失败记录
4. 新增 `GET /api/scan-runs/latest/failures`
5. 新增前端失败汇总页和首页入口
6. 补 README / UAT 文档说明
