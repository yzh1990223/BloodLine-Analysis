# 最近扫描输入持久化与自动回填设计

## 1. 背景

首页扫描控制面板当前支持手工填写：

- Repo 文件路径
- Java 源码目录
- MySQL DSN（预留）

但这些值在点击“重新扫描解析”后不会被持久化保存。页面刷新或重新打开后，用户需要重新输入，和“最近一次扫描状态”脱节，也不利于重复测试与 UAT。

## 2. 目标

本轮为最近一次扫描增加“输入持久化 + 自动回填”能力：

- 每次点击“重新扫描解析”时，把当前提交的输入保存到后端
- `/api/scan-runs/latest` 返回最近一次扫描对应的输入
- 首页扫描控制面板在首次加载最近扫描状态时，自动回填这些输入

## 3. 非目标

本轮不做：

- 多历史版本输入管理
- 用户维度的个性化表单记忆
- 仅依赖浏览器 `localStorage` 的本地存储方案
- 新增独立“输入模板”管理功能

## 4. 方案选择

### 方案 A：保存在后端 `scan_runs`

把最近扫描输入作为扫描记录的一部分持久化，在加载最近扫描状态时一起返回。

优点：

- 与“最近扫描状态”天然绑定
- 跨刷新、跨浏览器稳定
- 不会出现“界面记住了输入，但后端最近扫描并不是这次输入”的分裂状态

缺点：

- 需要扩展数据库模型和接口返回

### 方案 B：只保存在浏览器 `localStorage`

优点：

- 实现简单

缺点：

- 换浏览器或清缓存就丢失
- 无法与最近扫描记录对齐

本轮采用 **方案 A**。

## 5. 后端设计

### 5.1 数据模型

为 `scan_runs` 增加一个 JSON 字段：

- `inputs`

该字段保存用户本次扫描提交的输入，结构与 `POST /api/scan` 接收的 payload 保持一致，仅保留非空值。例如：

```json
{
  "repo_path": "/data/repository.xml",
  "java_source_root": "/data/java",
  "mysql_dsn": "mysql+pymysql://user:pass@localhost/dm",
  "metadata_databases": ["dm", "ods"]
}
```

### 5.2 接口行为

`POST /api/scan`

- 在创建扫描记录时，把请求 payload 的非空字段持久化到 `scan_runs.inputs`
- 返回体中的 `inputs` 保持现有行为

`GET /api/scan-runs/latest`

- 在 `scan_run` 节点下新增 `inputs`
- 当没有最近扫描记录时，仍返回 `{"scan_run": null}`

### 5.3 标准化规则

- 对空字符串做 `trim` 后为空的字段不持久化
- `metadata_databases` 只有在存在有效项时才持久化
- 不做额外加密或脱敏；第一版沿用现有“DSN 可提交到扫描接口”的边界

## 6. 前端设计

### 6.1 数据来源

扫描控制面板初始化时，调用 `/api/scan-runs/latest`：

- 如果返回 `scan_run.inputs`，则自动回填：
  - `repo_path`
  - `java_source_root`
  - `mysql_dsn`
- 未返回的字段保持为空

### 6.2 交互规则

- 自动回填只发生在首次加载最近扫描结果时
- 用户手工修改后，以本地输入为准
- 点击“重新扫描解析”成功后，表单保留当前输入，不需要清空

## 7. 测试

后端测试：

- 最近扫描记录返回 `inputs`
- `POST /api/scan` 会保存 `repo_path / java_source_root / mysql_dsn / metadata_databases`
- 空值字段不会被持久化

前端测试：

- 当最近扫描返回 `inputs` 时，表单自动回填
- 提交后请求 payload 仍只发送非空字段

## 8. 风险与控制

风险：

- 老数据库没有 `inputs` 字段
- 前端在异步加载后覆盖用户正在输入的值

控制：

- 增加 Alembic 迁移
- 只在首次加载最近扫描时进行回填，避免重复覆盖用户输入
