# 经验沉淀闭环设计

## 1. 背景

BloodLine Analysis 已经形成了较完整的治理体系与 Vibe Coding 工作流，但当前经验仍然散落在：

- 会话过程
- commit 历史
- 阶段总结
- 零散治理文档

这会导致一个问题：经验可以被讨论，但很难被稳定复用，更难形成“记录 -> 提炼 -> 回灌”的持续闭环。

## 2. 目标

建立一套轻量但可执行的经验沉淀闭环体系，让以下经验都能进入统一流程：

- bug / 故障 / 误判案例
- 实现与设计取舍
- UAT / 部署 / 运行经验
- 治理与 AI 协作经验

第一版目标不是构建知识库平台，而是跑通这条主链路：

`经验出现 -> 记录 -> 分类 -> 提炼 -> 回灌 -> 后续复用`

## 3. 非目标

第一版不做：

- 复杂标签系统
- 评分引擎
- 自动经验毕业状态机
- 数据库存储或检索平台
- 自动化 NLP 摘要

## 4. 总体结构

体系分为三层：

1. 原始经验记录层
2. 提炼归纳层
3. 回灌执行层

其中第三层最关键。经验条目只有在触发实际回灌动作后，才算形成闭环。

## 5. 分类模型

第一版使用 4 类分类：

- `incident`
- `implementation`
- `operation`
- `governance`

它们分别对应：

- 故障与误判
- 设计与实现取舍
- UAT / 部署 / 运行经验
- 提交、门禁、文档同步、AI 协作经验

## 6. 目录结构

```text
docs/experience/
├── README.md
├── incidents/
├── implementation/
├── operations/
├── governance/
├── digests/
└── indexes/
```

第一版目录中不放空说明文件，而是直接放：

- 一个主说明文档
- 一个索引
- 一个阶段 digest
- 每个分类至少一条真实样例经验

## 7. 单条经验模板

每条经验固定包含：

1. 类型
2. 背景
3. 现象
4. 根因
5. 影响
6. 处理
7. 可复用结论
8. 回灌动作

其中“可复用结论”和“回灌动作”是第一版最重要的字段。

## 8. 回灌机制

经验条目可回灌到 5 类目标：

- 文档
- 测试
- hooks / CI
- AGENTS
- backlog / 下一阶段需求

一条经验不要求同时命中全部 5 类，但至少应明确：

- 是否需要回灌
- 回灌到哪里
- 当前是否已落地

## 9. 治理接入方式

第一版通过以下方式接入现有治理体系：

- 在 `docs/governance/` 新增经验闭环基础文档
- 在 `README.md` 中增加经验体系入口
- 在 `AGENTS.md` 中增加“重大经验应沉淀到 docs/experience/”的入口规则
- 在 `tests/governance_smoke.sh` 中检查经验体系核心文档存在

第一版不把经验闭环直接接入 Git hooks 强阻断，只先进入：

- 文档入口
- 治理入口
- smoke 校验

## 10. 第一版样例范围

第一版直接沉淀至少 4 条真实经验：

- 路径带空格导致扫描失败
- 首页全量总览图不适合大图场景
- 双前端 dev server 导致验证混乱
- AI hooks 不应重复 superpowers

并在 digest 中做一次阶段性归纳。

## 11. 验证方式

第一版实现完成后，至少验证：

- `docs/experience/` 目录结构存在
- 经验主说明、索引、digest、样例条目存在
- `governance_smoke.sh` 已纳入经验体系核心文档检查
- `README.md` 与 `AGENTS.md` 已提供入口

## 12. 成功标准

第一版完成后，应达到：

- 仓库中有统一经验目录
- 有可执行模板
- 有索引
- 有真实样例
- 有治理接入点
- 有后续回灌方向

这样经验体系才不只是“多一个 docs 目录”，而是真正开始变成项目记忆的一部分。
